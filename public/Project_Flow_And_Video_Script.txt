Task 1: Complete Flow (Components, Packages, and Page-wise Explanation)
======================================================================

Packages used
-------------
1. react, react-dom
   - Core UI rendering.
2. react-router-dom
   - Routing and navigation between pages.
3. lucide-react
   - Icons used across the UI.
4. vite
   - Development server and production build tool.
5. tailwindcss (+ postcss, autoprefixer)
   - Utility-first styling.
6. eslint stack
   - Linting and code quality checks.


App startup flow
----------------
1. Entry point is src/main.jsx
2. ThemeProvider (src/context/ThemeContext.jsx) wraps the app for light/dark mode state.
3. AppRoutes (src/app/routes.jsx) handles all routes.
4. Global styles and design tokens are in src/index.css


Routing flow
------------
1. /         -> Signup page
2. /login    -> Login page
3. /dashboard -> Dashboard page
4. /scans and /scan/:id -> Scan Detail page


How components connect
----------------------
1. Layout components
   - AppLayout.jsx: overall shell (sidebar + content area)
   - Sidebar.jsx: navigation, profile area, logout dropdown
   - PageHeader.jsx: top action bar

2. Reusable UI components
   - Button.jsx: standard button variants
   - StatusChip.jsx: scan status labels
   - SeverityBadge.jsx: vulnerability severity badges
   - Skeleton.jsx: loading placeholders
   - PageTransition.jsx: page entry motion wrapper

3. Data and state
   - Mock scan data from src/data/data.js
   - Theme state from ThemeContext
   - Local page states for loading, filters, pagination, toasts, etc.


Page-by-page easy explanation
-----------------------------
Signup page (src/pages/Signup.jsx)
1. Split-screen UI: left info panel + right form panel.
2. Validates password length.
3. On submit, sets session flag (dashboard entry loader) and navigates to /dashboard.
4. Contains link to Login page.

Login page (src/pages/Login.jsx)
1. Same visual style as Signup page.
2. Takes email + password and validates basic input.
3. On success, sets same dashboard entry loader flag and navigates to /dashboard.
4. Contains link back to Signup.

Dashboard page (src/pages/Dashboard.jsx)
1. Uses AppLayout + PageHeader.
2. Reads and renders mock scan data.
3. Shows full-page entry loader when coming from Signup/Login.
4. Shows skeleton loaders while dashboard data resolves.
5. Includes search, status filter, column presets, and pagination.
6. Table rows are keyboard accessible; row click/Enter/Space opens scan details.

Scan Detail page (src/pages/ScanDetail.jsx)
1. Uses AppLayout + PageHeader.
2. Top section shows scan progress and scan steps.
3. Main section has activity/verification tabs + findings list.
4. Uses skeleton loading states and reusable badges/buttons.

Task 3: Functional Deep-Dive (Dark Mode, Global CSS, Search/Filter, Pagination, Page Updates)
===============================================================================================

How dark mode is made functional
--------------------------------
1. Theme state is centralized in src/context/ThemeContext.jsx
   - `theme` stores current mode (`light` or `dark`).
   - `toggleTheme()` switches mode and writes it to localStorage (`fenrir_shell_theme`).
   - `isSwitching` and `targetTheme` are used to show transition loaders.

2. App shell consumes this state in AppLayout.jsx
   - Root container gets `theme-dark` or `theme-light` class.
   - Because of this class, CSS variables automatically switch values.

3. Components using theme
   - Sidebar theme toggle button triggers `toggleTheme`.
   - Signup/Login page top-right theme toggle also triggers `toggleTheme`.
   - Theme persists after reload due to localStorage read on startup.


How global CSS is working
-------------------------
1. Global stylesheet is src/index.css
   - Defines root motion tokens and keyframes.
   - Defines shared shell variables (colors, borders, backgrounds, text).

2. Theme variable system
   - `.app-shell` defines light tokens.
   - `.app-shell.theme-dark` overrides the same tokens for dark mode.
   - Components use `var(--shell-...)` tokens, so theme switch updates UI globally.

3. Shared animations and behavior
   - Route and page entry animations (`route-screen-motion`, `page-transition-enter`).
   - Skeleton shimmer animation for loaders.
   - Reduced-motion media query disables heavy motion for accessibility.


How search, filter, and button functionalities work (Dashboard)
---------------------------------------------------------------
1. Search
   - Input updates `search` state on each change.
   - Filtering logic checks scan name and type with case-insensitive match.
   - Page is reset to first page when search changes.

2. Status filter button
   - Clicking filter cycles through: `all -> Completed -> Scheduled -> Failed`.
   - `activeFilter` state updates and list re-computes.
   - Toast confirms current filter.

3. Column preset button
   - Cycles view modes: `All Columns`, `Compact View`, `Risk View`.
   - `activeColumns` determines which table columns are rendered.

4. New scan button
   - Adds a new mock scan object at top of list.
   - Auto-generates ID and default scheduled values.
   - Toast confirms new scan creation.

5. Reusable button system
   - `Button.jsx` standardizes visual variants and focus states.
   - Used across top bars, controls, and actions.


How pagination works
--------------------
1. `pageSize` is fixed (15 rows per page).
2. `filteredScans` is computed first (after search/filter).
3. `totalPages` is calculated from filtered result count.
4. `clampedPage` prevents going beyond available pages.
5. `paginatedScans` slices data for current page only.
6. Prev/Next buttons update page safely with min/max bounds.
7. Pagination count text updates in real time based on current filtered data.


How page turn/update behavior works
-----------------------------------
1. Route transitions
   - `routes.jsx` wraps route rendering in a motion container keyed by pathname.
   - On route change, entry animation plays for the next screen.

2. Dashboard entry loader
   - Signup/Login sets a sessionStorage flag (`dashboard_entry_loader`) before navigate.
   - Dashboard reads that flag on mount and shows full-page loader briefly.
   - Flag is removed after read, so loader appears only once per auth entry.

3. Data loading/skeleton update
   - Dashboard and Scan Detail start with loading state (`isLoading = true`).
   - A timeout simulates API delay.
   - While loading, skeleton placeholders render.
   - After timeout, data sections replace skeletons automatically.

4. Scan row navigation update
   - Clicking a row or pressing Enter/Space navigates to `/scan/:id`.
   - This makes table navigation both mouse and keyboard friendly.
